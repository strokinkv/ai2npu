# Техническое задание: ai2npu

## 1. Общие сведения

### 1.1. Наименование

Рабочее название программного продукта: **ai2npu**.

Имя исполняемого файла: `ai2npu.exe`.

Имя Windows-службы: `ai2npuService`.

Отображаемое имя Windows-службы: `ai2npu Service`.

### 1.2. Назначение

`ai2npu` — локальная Windows-служба для запуска AI-моделей на Intel NPU через OpenVINO Runtime.

Программа должна предоставлять локальный HTTP API, совместимый с основными форматами OpenAI API, для следующих сценариев:

- распознавание речи через Whisper;
- перевод аудио через Whisper;
- построение embedding-векторов;
- получение списка доступных моделей;
- получение состояния службы;
- получение последних строк лог-файла для локальной диагностики.

Генерация текста через небольшие LLM не входит в первую версию и должна рассматриваться как направление будущего развития.

Основной сценарий использования — локальный запуск на Windows 11 на устройстве с процессором **Intel Core Ultra 7 258V** или совместимым Intel NPU.

### 1.3. Целевая платформа

- Операционная система: Windows 11.
- Архитектура: x64.
- Аппаратное ускорение: Intel NPU.
- Runtime: последняя актуальная версия OpenVINO 2026.2
- Язык реализации: Rust.

### 1.4. Основные ограничения

1. Инференс моделей должен выполняться только на устройстве `NPU`.
2. Fallback на CPU или GPU запрещён.
3. API должен слушать только `127.0.0.1`.
4. HTTPS не требуется.
5. Аутентификация не требуется.
6. UI не требуется.
7. Настройка выполняется через один TOML-конфиг.
8. Сервис должен работать как один процесс.
9. Модели должны загружаться в память при первом обращении и выгружаться после периода неактивности, если не указано иное.
10. Тела пользовательских запросов не должны записываться в лог.

## 2. Размещение файлов

### 2.1. Исполняемые файлы

Исполняемые файлы и неизменяемые ресурсы устанавливаются в:

`C:\Program Files\ai2npu\`

Содержимое директории:

- `ai2npu.exe`;
- runtime-библиотеки;
- необходимые DLL;
- документация;
- примеры конфигурации;
- вспомогательные ресурсы, необходимые для запуска.

### 2.2. Конфигурация

Основной конфигурационный файл располагается в:

`C:\ProgramData\ai2npu\config.toml`

Использование `ProgramData` выбрано вместо размещения конфига рядом с `.exe`, так как при установке в `Program Files` изменение конфигурации рядом с исполняемым файлом может требовать повышенных прав.

### 2.3. Логи

Логи располагаются в:

`C:\ProgramData\ai2npu\logs\`

Основной лог-файл:

`C:\ProgramData\ai2npu\logs\ai2npu.log`

Логи ротируются и удаляется история согласно конфигурации.

### 2.4. Модели

Модели располагаются в:

`C:\ProgramData\ai2npu\models\`

Текущая структура установки:

- `C:\ProgramData\ai2npu\models\OpenVINO\whisper-large-v3-turbo-int8-ov\`
- `C:\ProgramData\ai2npu\models\strokinkv\bge-m3-int8-ov\`

## 3. Windows-служба

### 3.1. Параметры службы

- Service name: `ai2npuService`.
- Display name: `ai2npu Service`.
- Startup type: `Automatic`.
- После установки служба должна стартовать автоматически.
- После перезагрузки ПК служба должна запускаться автоматически.

### 3.2. Пользователь службы

Рекомендуемый пользователь службы: `LocalService`.

Требования к правам:

- чтение `C:\Program Files\ai2npu\`;
- чтение и запись `C:\ProgramData\ai2npu\`;
- доступ к OpenVINO Runtime;
- доступ к Intel NPU device.

### 3.3. Управление службой

Служба должна управляться через CLI-команды `ai2npu.exe` и стандартные средства Windows Service Control Manager.

## 4. CLI

`ai2npu.exe` должен поддерживать следующие команды:

| Команда | Назначение |
| --- | --- |
| `ai2npu.exe run --config <path>` | Запуск в консольном режиме для отладки. |
| `ai2npu.exe install-service` | Установка Windows-службы. |
| `ai2npu.exe uninstall-service` | Удаление Windows-службы. |
| `ai2npu.exe start-service` | Запуск службы. |
| `ai2npu.exe stop-service` | Остановка службы. |
| `ai2npu.exe restart-service` | Рестарт службы, в том числе после изменения конфига. |
| `ai2npu.exe validate-config` | Проверка корректности TOML-конфига. |
| `ai2npu.exe check-npu` | Проверка доступности Intel NPU для использования |
| `ai2npu.exe list-models` | Вывод моделей, описанных в конфиге. |
| `ai2npu.exe version` | Вывод версии программы, OpenVINO Runtime и сборочной информации. |

Команда `convert-model` в первой версии не требуется, так как подготовленный набор моделей поставляется вместе с Inno Setup установщиком.

Если `validate-config` вызывается без `--config`, CLI должен искать конфиг по умолчанию:

1. `C:\ProgramData\ai2npu\config.toml`;

В режиме разработки команда:

`ai2npu.exe run --config config.example.toml`

должна запускаться из корня проекта и корректно разрешать относительные пути к директории `models`.

## 5. Установщик

### 5.1. Формат установщика

Inno Setup

### 5.2. Требования к установщику

Установщик должен:

1. Устанавливать файлы программы в `C:\Program Files\ai2npu\`.
2. Создавать директорию `C:\ProgramData\ai2npu\`, если она отсутствует.
3. Создавать директорию `C:\ProgramData\ai2npu\logs\`, если она отсутствует.
4. Создавать директорию `C:\ProgramData\ai2npu\models\`, если она отсутствует.
5. Создавать файл конфигурации из примера `C:\ProgramData\ai2npu\config.toml`, если файл отсутствует.
6. Не перезаписывать существующий `config.toml` при обновлении.
7. Устанавливать службу `ai2npuService`.
8. Назначать службе тип запуска `Automatic`.
9. Запускать службу автоматически сразу после установки.
10. Поддерживать silent install.
11. Поддерживать silent uninstall.
12. Обеспечивать корректное удаление службы при uninstall.
13. Добавлять `C:\Program Files\ai2npu` в системный `PATH`, чтобы `ai2npu.exe` можно было вызывать без полного пути из новых терминальных сессий.
14. Настраивать Windows Service Recovery для `ai2npuService`: restart после crash с задержками `5s`, `5s`, `30s`, reset period `86400s`.
15. Позволять в интерактивном установщике выбрать устанавливаемые модели: `BAAI/bge-m3`, `openai/whisper-large-v3-turbo` или не устанавливать модели вообще. По умолчанию обе модели выбраны.
16. Не удалять пользовательский `config.toml` и модели без явного согласия пользователя.

## 6. Конфигурация

### 6.1. Общие требования

Формат конфигурации: TOML.

Основной конфиг:

`C:\ProgramData\ai2npu\config.toml`

Конфиг читается при старте службы.

Горячая перезагрузка конфига в первой версии не требуется. После изменения конфига пользователь должен выполнить рестарт службы:

`ai2npu.exe restart-service`

### 6.2. Рекомендуемая структура конфига

Конфиг должен содержать следующие секции:

- `[server]`
- `[queue]`
- `[logging]`
- `[[models]]`

### 6.3. Параметры `[server]`

| Параметр | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `host` | string | `127.0.0.1` | Интерфейс, на котором слушает HTTP API. |
| `port` | integer | `9555` | Порт по умолчанию HTTP API. |
| `request_body_limit_mb` | integer | `100` | Максимальный размер тела HTTP-запроса. |
| `thread_count` | integer | `16` | Количество HTTP worker threads. Должно быть больше `max_pending_requests + 1`, чтобы очередь могла заполняться и возвращать `429 Too Many Requests` под параллельной нагрузкой. |

### 6.4. Параметры `[queue]`

| Параметр | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `max_pending_requests` | integer | `50` | Максимальное количество ожидающих запросов. Выполняющийся запрос не учитывается. |
| `default_timeout_sec` | integer | `600` | Таймаут ожидания запроса в очереди. |

### 6.5. Параметры `[logging]`

| Параметр | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `level` | string | `info` | Уровень логирования. |
| `directory` | string | `C:\ProgramData\ai2npu\logs` | Директория логов. |
| `format` | string | `text` | Формат логов. В первой версии используется текст. |
| `max_file_size_mb` | integer | `10` | Максимальный размер одного лог-файла. |
| `max_files` | integer | `10` | Максимальное количество ротируемых лог-файлов. |

Поддерживаемые уровни логирования:

- `trace`
- `debug`
- `info`
- `warn`
- `error`

### 6.6. Параметры `[[models]]`

Для каждой модели должны задаваться:

| Параметр | Тип | Описание |
| --- | --- | --- |
| `id` | string | Имя модели во внешнем API. Должно совпадать с именем на Hugging Face. |
| `type` | string | Тип модели первой версии: `whisper`, `embedding`. Значение `chat` зарезервировано для будущей LLM-функциональности. |
| `path` | string | Локальный путь к директории модели. |
| `enabled` | boolean | Включена ли модель. |
| `preload` | boolean | Загружать ли модель при старте службы. |
| `queue_timeout_sec` | integer | Таймаут ожидания в очереди для данной модели. |

Загруженная модель остаётся в памяти до завершения работы службы или явной выгрузки через `POST /admin/models/unload` (`ai2npu unload`); автоматической выгрузки по простою нет.

Для embedding-моделей дополнительно:

| Параметр | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `normalize` | boolean | `true` | Выполнять L2-normalization embedding-векторов. |

Для Whisper-моделей дополнительно:

| Параметр | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `max_audio_duration_sec` | integer | `1800` | Максимальная длительность аудио, 30 минут. |

## 7. Модели

### 7.1. Whisper

Основная модель:

`openai/whisper-large-v3-turbo`

Назначение:

- транскрибация аудио;
- перевод аудио.

Endpoint'ы:

- `POST /v1/audio/transcriptions`
- `POST /v1/audio/translations`

Поддерживаемый формат входного файла:

- `wav`.

Сервис не выполняет конвертацию, downmix или resample аудио в первой версии.

Входной WAV должен быть заранее подготовлен клиентом в обязательном формате:

- RIFF WAV;
- PCM format tag `1`;
- mono;
- 16 kHz;
- 16-bit signed little-endian, `pcm_s16le`.

Float WAV, stereo WAV, WAV с частотой дискретизации, отличной от 16 kHz, и любые сжатые аудиоформаты не поддерживаются. Если WAV не соответствует обязательному формату, сервис должен вернуть ошибку `400 Bad Request` с OpenAI-like описанием причины.

Внутри сервиса должны выполняться только:

- чтение WAV;
- валидация WAV header/format;
- проверка длительности;
- нормализация аудио, если она требуется конкретным Whisper pipeline;
- подготовка входных тензоров для Whisper.

FFmpeg как внешняя зависимость не используется.

Максимальная длительность аудио, согласно конфигурации, по умолчанию 30 минут.

Если длительность аудио превышает выставленный лимит, сервис должен вернуть ошибку.

Поддерживаемые значения `response_format` должны соответствовать OpenAI Audio API для Whisper-like сценариев:

- `json`;
- `text`;
- `srt`;
- `verbose_json`;
- `vtt`.

Значение по умолчанию: `json`.

Для `json` и `verbose_json` ответ должен иметь `Content-Type: application/json`. Для `text`, `srt` и `vtt` ответ должен возвращаться как текстовый body без JSON-обёртки.

Минимальный `json` ответ:

```json
{
  "text": "..."
}
```

Минимальный `verbose_json` ответ:

```json
{
  "text": "...",
  "language": "...",
  "duration": 0.0,
  "segments": []
}
```

Для `/v1/audio/transcriptions` параметр `timestamp_granularities` применяется только при `response_format = "verbose_json"`. Поддерживаемые значения:

- `segment`;
- `word`.

Для `/v1/audio/translations` параметр `timestamp_granularities` не поддерживается.

Значение `diarized_json` в первой версии не поддерживается, так как выбранная модель `openai/whisper-large-v3-turbo` не выполняет diarization. При неподдерживаемом `response_format` сервис должен вернуть `400 Bad Request` с OpenAI-like ошибкой.

Поддерживаемые параметры запроса:

- `model`
- `file`
- `language`
- `prompt`
- `response_format`
- `temperature`
- `timestamp_granularities`

`temperature` принимается в диапазоне `0.0`–`1.0` и передаётся в generation config Whisper-пайплайна; при отсутствии используется температура модели по умолчанию, при выходе за диапазон возвращается `400 Bad Request`.

Неподдерживаемые параметры допускается принимать и игнорировать с записью сообщения уровня `debug`.

Whisper pipeline в первой версии должен использовать подготовленный OpenVINO GenAI bundle и запускаться через `WhisperPipeline` на устройстве `NPU`. Ручная реализация encoder/decoder generation loop не требуется, если OpenVINO GenAI Runtime поддерживает поставляемый bundle.

### 7.2. Embeddings

Основная модель:

`BAAI/bge-m3`

Назначение:

- построение embedding-векторов для русского и английского текста.

Endpoint:

- `POST /v1/embeddings`

Требования:

- поддерживать один входной текст;
- поддерживать batch-запросы через массив строк;
- не вводить жёсткий лимит на количество строк в batch на уровне ТЗ;
- не вводить жёсткий лимит длины строки на уровне ТЗ;
- фактические ограничения определяются моделью и доступными ресурсами;
- по умолчанию выполнять L2-normalization;
- позволять отключить нормализацию через конфиг модели;
- tokenization должна выполняться через OpenVINO tokenizer bundle, поставляемый вместе с моделью;
- модельный inference должен выполняться на устройстве `NPU`;
- сервис должен использовать выход `sentence_embedding`;
- ожидаемая длина embedding-вектора для `BAAI/bge-m3`: `1024`.

## 8. HTTP API

### 8.1. Общие требования

API должен быть совместим с основными форматами OpenAI API.

Базовый адрес по умолчанию:

`http://127.0.0.1:9555`

Аутентификация не требуется.

HTTPS не требуется.

CORS не требуется в первой версии.

Rate limiting не требуется.

### 8.2. Обязательные endpoint'ы

| Метод | Путь | Назначение |
| --- | --- | --- |
| `GET` | `/v1/models` | Список доступных моделей. |
| `POST` | `/v1/audio/transcriptions` | Распознавание речи. |
| `POST` | `/v1/audio/translations` | Перевод аудио. |
| `POST` | `/v1/embeddings` | Построение embeddings. |
| `GET` | `/health` | Состояние службы. |
| `GET` | `/logs` | Последние строки основного log-файла для локальной диагностики. |

### 8.3. `/v1/models`

Endpoint должен возвращать модели, заданные в конфиге и включённые через `enabled = true`.

Для каждой модели возвращаются:

- `id`;
- `object = "model"`;
- `created`;
- `owned_by`.

Имена моделей должны совпадать с именами Hugging Face:

- `openai/whisper-large-v3-turbo`;
- `BAAI/bge-m3`.

### 8.4. `/health`

Endpoint должен возвращать состояние службы.

Минимальный набор данных:

- служба запущена;
- OpenVINO Runtime доступен;
- NPU обнаружен;
- список моделей из конфига;
- список загруженных моделей;
- размер очереди;
- наличие текущего выполняемого запроса без пользовательских данных;
- uptime.

Endpoint не защищается, так как сервис слушает только `127.0.0.1`.

### 8.5. `/logs`

Endpoint должен возвращать последние строки `C:\ProgramData\ai2npu\logs\ai2npu.log` в JSON.

Параметры query string:

| Параметр | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `lines` | integer | `200` | Количество последних строк. Допустимый диапазон: `1..1000`. |

Формат ответа:

```json
{
  "path": "C:\\ProgramData\\ai2npu\\logs\\ai2npu.log",
  "lines": ["..."]
}
```

Endpoint не защищается, так как сервис слушает только `127.0.0.1`.

### 8.6. Ошибки API

Ошибки должны возвращаться в OpenAI-like формате.

Минимальные поля ошибки:

- `error.message`;
- `error.type`;
- `error.code`.

Минимальный набор стабильных `error.code`:

- `invalid_request`;
- `model_not_found`;
- `invalid_audio_format`;
- `payload_too_large`;
- `queue_full`;
- `queue_timeout`;
- `npu_unavailable`;
- `openvino_unavailable`;
- `model_load_failed`;
- `inference_failed`;
- `internal_error`.

Рекомендуемые HTTP-коды:

| Код | Сценарий |
| --- | --- |
| `400` | Некорректный запрос. |
| `404` | Модель не найдена или endpoint не найден. |
| `413` | Тело запроса больше 100 MB. |
| `429` | Очередь заполнена. |
| `500` | Внутренняя ошибка. |
| `503` | NPU или OpenVINO Runtime недоступны. |
| `504` | Таймаут ожидания в очереди. |

## 9. Очередь запросов

### 9.1. Общие требования

Очередь должна быть:

- общей на весь сервис;
- FIFO;
- с лимитом 50 ожидающих запросов;
- с настраиваемым таймаутом ожидания;
- с поддержкой отмены запроса при отключении клиента.

Выполняющийся запрос не считается ожидающим.

Если один запрос выполняется и 50 запросов ожидают, следующий запрос должен получить `429 Too Many Requests`.

### 9.2. Таймаут ожидания

Таймаут ожидания по умолчанию:

- 600 секунд.

Если запрос ждёт в очереди дольше таймаута, сервис должен вернуть ошибку `504 Gateway Timeout` или OpenAI-like ошибку с кодом таймаута.

### 9.3. Параллельное выполнение

Не допускается

## 10. Жизненный цикл моделей

### 10.1. Lazy loading

Модели должны загружаться лениво:

1. Сервис стартует.
2. Конфиг читается.
3. Модели не загружаются, если для них не указано `preload = true`.
4. При первом обращении к модели она загружается в память и подготавливается для инференса на `NPU`.
5. После завершения запроса модель остаётся загруженной.
6. Модель остаётся в памяти до остановки службы или явной выгрузки через `POST /admin/models/unload` (`ai2npu unload`); выгрузки по простою нет.

### 10.2. Ошибка загрузки модели

Если модель не удалось загрузить:

- текущий запрос должен получить ошибку;
- повторная попытка загрузки выполняется при следующем обращении;
- ошибка должна быть записана в лог.

### 10.4. Изменение конфига

Горячая перезагрузка конфига в первой версии не требуется.

Изменения конфига применяются после рестарта службы:

`ai2npu.exe restart-service`

## 11. Логирование

### 11.1. Общие требования

Логи пишутся в файл.

Формат логов:

- текстовый.

Тела запросов логировать запрещено.

### 11.2. Уровни логирования

Поддерживаемые уровни:

- `trace`
- `debug`
- `info`
- `warn`
- `error`

### 11.3. Ротация логов

Ротация должна выполняться:

- по размеру файла;
- по количеству файлов.

Значения по умолчанию:

- максимальный размер файла: 10 MB;
- максимальное количество файлов: 10.

### 11.4. События для логирования

В лог должны попадать:

- старт службы;
- остановка службы;
- рестарт службы;

- список найденных устройств;
- чтение конфига;
- ошибки конфига;
- загрузка модели;
- выгрузка модели;
- ошибки OpenVINO;
- переполнение очереди;
- таймауты очереди;
- отмена запроса клиентом;
- длительность ожидания в очереди;
- длительность инференса;
- длительность загрузки модели;
- HTTP-коды ответов.

## 13. Подготовка моделей

### 13.1. Общие требования

В первой версии команда `convert-model` не требуется, так как подготовленный набор моделей поставляется вместе с установщиком exe в виде заархивированных файлов.

Для разработки и сборки дистрибутива требуется pipeline подготовки моделей.

Pipeline должен:

1. Скачивать модели с Hugging Face.
2. Конвертировать модели в формат, пригодный для OpenVINO Runtime.
3. Оптимизировать и/или квантовать модели, если это требуется.
4. Проверять загрузку модели через OpenVINO Runtime.
5. Проверять запуск модели на устройстве `NPU`.
6. Создавать для каждой модели архивный файл, который будет использоваться установщиком.
7. Фиксировать версии зависимостей.

### 13.2. Приоритетный подход

Перед разработкой собственного конвертера необходимо исследовать возможность использования или адаптации `export_model.py` из OpenVINO Model Server:

`https://github.com/openvinotoolkit/model_server/blob/main/demos/common/export_models/export_model.py`

Если данный скрипт покрывает нужные сценарии для Whisper и embeddings, его следует использовать как основу pipeline подготовки моделей. Поддержка LLM должна быть исследована отдельно на будущем этапе.

Если возможностей скрипта недостаточно, допускается разработать собственный Python pipeline поверх:

- OpenVINO tooling;
- Optimum Intel;
- Hugging Face Transformers;
- Hugging Face Hub.

### 13.3. Модели первой версии

Модели, которые должны быть проверены в первую очередь:

- `openai/whisper-large-v3-turbo`;
- `BAAI/bge-m3`.

### 13.4. Контракт директорий моделей

Для `BAAI/bge-m3` директория модели должна содержать минимум:

- `model.xml`;
- `model.bin`;
- `tokenizer.json`;
- `config.json`;
- `sentencepiece.bpe.model`.

Модельный граф принимает:

- `input_ids` формы `[1,512]`, тип `int64`;
- `attention_mask` формы `[1,512]`, тип `int64`.

Сервис использует выход `sentence_embedding` формы `[1,1024]`.

Для `openai/whisper-large-v3-turbo` директория модели должна содержать минимум:

- `config.json`;
- `generation_config.json`;
- `preprocessor_config.json`;
- `tokenizer.json`;
- `openvino_encoder_model.xml`;
- `openvino_encoder_model.bin`;
- `openvino_decoder_model.xml`;
- `openvino_decoder_model.bin`;
- `openvino_tokenizer.xml`;
- `openvino_tokenizer.bin`;
- `openvino_detokenizer.xml`;
- `openvino_detokenizer.bin`.

Whisper bundle должен быть совместим с OpenVINO GenAI `WhisperPipeline`.

## 14. Выбранный стек реализации

### 14.1. Язык и сборка

- Rust stable.
- Cargo workspace с одним основным binary crate `ai2npu`.
- Целевая платформа сборки: `x86_64-pc-windows-msvc`.

### 14.2. Runtime и инференс

- OpenVINO Runtime.
- OpenVINO 2026.2.
- Основной доступ к OpenVINO Runtime из Rust: через OpenVINO C API и тонкую Rust FFI-обёртку, если существующие Rust crates не обеспечивают стабильную поддержку OpenVINO 2026.2 и устройства `NPU`.
- Для Whisper в первой версии используется OpenVINO GenAI `WhisperPipeline`. Если для OpenVINO GenAI нет готового Rust binding, допускается реализовать минимальную C/C++ FFI-обёртку вокруг `ov::genai::WhisperPipeline`.
- Для embeddings используется OpenVINO Runtime: tokenization выполняется Rust tokenizer из `tokenizer.json`, inference использует `model.xml` / `model.bin`.
- Модельный inference должен компилироваться и выполняться только на устройстве `NPU`.
- Fallback на `CPU`, `GPU`, `AUTO`, `HETERO` и другие устройства запрещён.
- Если `NPU` отсутствует или модель не компилируется на `NPU`, сервис возвращает ошибку и не пытается выполнить модель на другом устройстве.
- Pre/post-processing, включая чтение WAV, проверку WAV и tokenization через OpenVINO tokenizer, не считается модельным inference. Эти операции могут выполняться на CPU, если это требуется используемым runtime-компонентом.

### 14.3. HTTP-сервер

- `tokio`.
- `axum`.
- `tower-http` для лимита тела запроса и служебных middleware, если требуется.

### 14.4. JSON

- `serde`.
- `serde_json`.

### 14.5. TOML

- `toml`.
- `serde` для десериализации и валидации конфигурации.

### 14.6. Логирование

- `tracing`.
- `tracing-subscriber`.
- Ротация файлов реализуется в сервисе или через совместимый Rust crate, если он поддерживает ротацию по размеру и количеству файлов на Windows.

### 14.7. CLI

- `clap`.
- Интеграция с Windows Service Control Manager через crate `windows-service`.

### 14.8. Тестирование

- Unit-тесты через стандартный Rust test framework.
- HTTP integration tests для API-контрактов.
- Конфигурационные тесты для `validate-config`.
- NPU smoke tests для проверки OpenVINO Runtime, загрузки моделей и запрета fallback.

### 14.9. Audio processing

В первой версии сервис не выполняет конвертацию WAV, downmix или resample. Отдельные audio resampling библиотеки, такие как `soxr` или `libsamplerate`, не используются.

Требуется только минимальный WAV reader/validator для проверки обязательного входного формата.

## 15. Архитектура

### 15.1. Компоненты

Несмотря на то, что программа должна работать как один процесс, внутри рекомендуется логически разделить компоненты:

1. `ServiceHost` — интеграция с Windows Service Control Manager.
2. `ConsoleHost` — запуск в консольном режиме.
3. `ConfigManager` — загрузка и валидация TOML-конфига.
4. `HttpServer` — HTTP API.
5. `RequestRouter` — маршрутизация API-запросов.
6. `QueueManager` — общая FIFO-очередь задач.
7. `ModelManager` — загрузка, хранение, выгрузка моделей.
8. `OpenVinoBackend` — работа с OpenVINO Runtime и NPU.
9. `WhisperPipeline` — обработка аудио и Whisper-инференс.
10. `EmbeddingPipeline` — tokenization, inference и постобработка embeddings.
11. `Logger` — логирование и ротация логов.

Компонент `ChatPipeline` не входит в первую версию и может быть добавлен на будущем этапе LLM-функциональности.

### 15.2. Основной поток обработки запроса

1. Клиент отправляет HTTP-запрос на `127.0.0.1:9555`.
2. `HttpServer` принимает запрос.
3. `RequestRouter` валидирует endpoint и базовый формат запроса.
4. Проверяется размер тела запроса.
5. Проверяется наличие модели в конфиге.
6. Запрос помещается в общую FIFO-очередь.
7. Если очередь заполнена, возвращается `429 Too Many Requests`.
8. Когда запрос доходит до выполнения, `ModelManager` проверяет, загружена ли модель.
9. Если модель не загружена, она загружается через `OpenVinoBackend` на `NPU`.
10. Выполняется инференс.
11. Результат преобразуется в OpenAI-compatible формат.
12. Ответ отправляется клиенту.
13. Логи обновляются.
14. Модель остаётся загруженной до idle timeout или постоянно, если idle timeout = 0.

## 16. Нефункциональные требования

### 16.1. Надёжность

- Ошибка одной модели не должна останавливать службу.
- Некорректный запрос не должен приводить к падению процесса.
- Ошибка инференса должна возвращаться клиенту в виде OpenAI-like ошибки.
- Ошибки OpenVINO должны логироваться.

### 16.2. Безопасность

- API должен слушать только `127.0.0.1`.
- Аутентификация не требуется.
- Тела запросов не логируются.
- Rate limiting не требуется.

### 16.3. Производительность

На уровне первой версии фиксируются функциональные требования, а точные latency/SLA должны быть уточнены после proof-of-concept на Intel Core Ultra 7 258V.

### 16.4. Обслуживаемость

- Конфиг должен быть человекочитаемым TOML.
- Ошибки конфигурации должны быть понятными.
- Должна быть CLI-команда `validate-config`.
- Должна быть CLI-команда `restart-service`.
- Должен быть режим запуска `run` для отладки в консоли.

## 19. Критерии приёмки первой версии

Первая версия считается принятой, если выполнены следующие условия:

1. Программа устанавливается через Inno Setup `.exe` установщик на Windows 11.
2. Установщик создаёт необходимые директории в `ProgramData`.
3. Установщик создаёт службу `ai2npuService` с типом запуска `Automatic`.
4. Служба запускается сразу после установки.
5. Служба запускается после перезагрузки ПК.
6. Интерактивный установщик позволяет выбрать `BAAI/bge-m3`, `openai/whisper-large-v3-turbo` или не устанавливать модели. По умолчанию обе модели выбраны.
7. `ai2npu.exe run` запускает программу в консольном режиме.
8. `ai2npu.exe validate-config` проверяет конфиг.
9. `ai2npu.exe restart-service` перезапускает службу.
10. API слушает только `127.0.0.1:9555`.
11. `GET /v1/models` возвращает список включенных моделей из конфига.
12. `GET /health` возвращает состояние службы, OpenVINO и NPU.
13. `/v1/embeddings` принимает OpenAI-like запрос и возвращает embedding-вектор.
14. `/v1/audio/transcriptions` принимает WAV до 30 минут и возвращает ответ в форматах `json`, `text`, `srt`, `verbose_json`, `vtt`.
15. `/v1/audio/translations` принимает WAV до 30 минут и возвращает ответ в форматах `json`, `text`, `srt`, `verbose_json`, `vtt`.
16. Модели загружаются при первом обращении.
17. Модель остаётся загруженной до остановки службы или явной выгрузки через `POST /admin/models/unload` (`ai2npu unload`).
18. При заполнении очереди возвращается `429 Too Many Requests`.
19. При превышении размера тела запроса возвращается `413 Payload Too Large`.
20. Тела запросов не пишутся в лог.
21. Логи ротируются.
22. Fallback на CPU/GPU не выполняется.
23. При невозможности запуска модели на NPU возвращается ошибка.
